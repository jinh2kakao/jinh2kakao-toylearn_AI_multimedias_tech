## 프롬프트 유형과 선택 이유
1. [핵심 기반] 3. 역할 부여 (Role-Playing)
- 선택 이유: 이 문제는 "아이디어를 내줘"가 아닌 "개발 명세를 만들어줘"라는 전문적인 요청입니다. AI에게 "10년 차 시니어 PM 또는 테크 리드(Tech Lead)" 역할을 부여해야 합니다. 이 역할은 주니어 PM이 놓치기 쉬운 비기능적 요구사항, DB 스키마, API 엔드포인트 등 개발자가 실제로 궁금해하는 기술적 세부 사항을 꿰뚫어 볼 수 있습니다.

- 효과: "깔끔한 대시보드"라는 말을 듣고 "아, 그건 일일 칼로리, 주간 트렌드 차트, 영양소 밸런스(탄/단/지) 도넛 차트가 필요한 프론트엔드 요구사항이군요"라고 기술적으로 '번역'해주는 전문가의 답변을 유도합니다.

2. [논리 엔진] 6. 체인 오브 스루트 (Chain of Thought, CoT)
- 선택 이유: 훈련 초점인 **[구조적 분석 및 세분화]**를 수행하는 가장 강력한 도구입니다. AI가 "식단을 입력하면 칼로리를 계산한다"는 단순한 문장을 논리적 단계에 따라 분해하도록 강제합니다.

- 효과: AI가 스스로 '① 사용자가 누구인가? (User Story) → ② 어떤 화면이 필요한가? (Pages) → ③ 각 화면은 어떻게 움직여야 하는가? (Front-end) → ④ 그 데이터를 누가 처리하는가? (Back-end) → ⑤ 데이터는 어디에 저장되는가? (Data)'라는 체계적인 사고 과정을 거쳐 결과물을 도출하게 만듭니다.

3. [출력 통제] 4. 제약 조건 (Constraint-Based)
- 선택 이유: 개발팀과 디자이너에게 필요한 결과물은 산문형 에세이가 아닌, **체크리스트 형태의 명확한 '문서'**입니다. 프롬프트에 Front-end, Back-end, UX/UI, Data 등 구체적인 카테고리(목차)를 명시하여 AI가 생성할 답변의 구조를 강력하게 제어해야 합니다.

- 효과: AI의 답변이 흩어지는 것을 방지하고, 주니어 PM이 미팅에 그대로 들고 갈 수 있는 구조화된 '요구사항 명세서(PRD/SRS)' 포맷으로 결과물을 정제합니다.

## 작성 위해 추가 필요 요건
1. 핵심 타겟 유저 (Target User): "깔끔한 대시보드"는 30대 남성 보디빌더와 20대 여성 다이어터에게 완전히 다르게 해석됩니다.

2. 핵심 가치 제안 (Unique Selling Proposition, USP): 이 서비스가 MyFitnessPal 같은 기존 앱과 다른 점은 무엇인가? (예: "국내 배달 음식 DB 특화", "AI 자동 인식" 등)

## 가상 문제 상황
1. 가상 문제 상황 (Scenario Challenge)
페르소나 (Persona):
직무: 헬스케어 스타트업 '웰니스 핏(Wellness Fit)'의 주니어 PM (Product Manager)
성향: 아이디어는 많지만, 이를 개발자나 디자이너가 이해할 수 있는 기술적 언어로 정리하는 데 어려움을 겪고 있음.
배경 (Background):
당신은 개인 맞춤형 영양 관리 웹 서비스인 '뉴트리 가이드(Nutri-Guide)'의 MVP(최소 기능 제품) 웹 프로토타입을 기획 중입니다.
일주일 뒤, 개발팀 및 디자인팀과 함께 '프로토타입 제작 킥오프 미팅'이 예정되어 있습니다.
현재는 "사용자가 식단을 입력하면 칼로리를 계산해준다", "깔끔한 대시보드가 있었으면 좋겠다" 정도의 추상적인 아이디어만 나열된 상태입니다.
핵심 문제 (Problem):
개발 리더로부터 **"기능 명세가 너무 모호해서 DB 설계나 프론트엔드 구조를 잡을 수 없다"**는 피드백을 받았습니다.
구체적인 필수 기능 리스트(Feature List), 사용자 흐름(User Flow), 데이터 요구사항이 정의되지 않으면 프로젝트 착수가 무기한 연기될 위기입니다.
해결 과제 (Task):
추상적인 아이디어를 개발자와 디자이너가 즉시 작업할 수 있는 수준의 **'웹 프로토타입 제작을 위한 상세 요구사항 리스트'**로 변환해야 합니다.
기능적 요구사항(회원가입, 식단 입력, 리포트 출력 등)과 비기능적 요구사항(보안, 로딩 속도 등), 그리고 필요한 페이지 구조를 빠짐없이 도출해내야 합니다.

2. 훈련 초점 (Training Focus)
이 문제는 [구조적 분석 및 세분화 (Structural Breakdown)] 능력을 평가.
단순히 "아이디어를 내줘"가 아니라, 모호한 요청을 **체계적인 카테고리(Front-end, Back-end, UX/UI 등)**로 나누어 구체화하는 프롬프팅 스킬이 필요.

## 프롬프트 실행 결과

---

**제목: '뉴트리 가이드' MVP 웹 프로토타입 요구사항 명세서 (v0.1)**

**1. 사용자 Epics 및 핵심 User Stories**

* **Epic 1: 회원 관리**
    * (User Story 1-1) As a 다이어트 초보 20대 여성, I want to sign up easily using my email, so I can start logging my food.
* **Epic 2: 식단 기록**
    * (User Story 2-1) As a user, I want to search for Korean delivery food (e.g., '뿌링클 콤보', '엽기 떡볶이') and log it easily, so I can get accurate calorie counts.
    * (User Story 2-2) As a user, I want to log food by meal (breakfast, lunch, dinner), so I can track my eating patterns.
* **Epic 3: 리포트/대시보드**
    * (User Story 3-1) As a user, I want to see my total calorie intake for the day at a glance, so I know if I'm within my goal.
    * (User Story 3-2) As a user, I want to see a simple breakdown of my macros (carbs, protein, fat), so I can see if I'm eating balanced meals.

**2. 핵심 페이지 구조 (User Flow)**

* `P-1: 랜딩/로그인 페이지` (회원가입 버튼 포함)
* `P-2: 회원가입 페이지` (이메일, 비밀번호)
* `P-3: 메인 대시보드` (로그인 후 첫 화면. 리포트 + 식단 입력 버튼)
* `P-4: 식단 입력 모달(Modal)` (대시보드에서 '아침/점심/저녁' 버튼 클릭 시 팝업)

**3. 상세 기능 명세 (Structural Breakdown)**

**3-1. Front-end / UX/UI 요구사항**
* **[공통]** 모든 페이지는 모바일 반응형 웹(Mobile-First)으로 제작되어야 함.
* **[P-3: 대시보드]** '깔끔한 대시보드'를 다음과 같이 구체화함:
    * (상단) 날짜 선택기 (Date Picker) (좌우 화살표로 날짜 이동)
    * (중앙) 오늘 섭취한 총칼로리를 보여주는 **'도넛 차트'** 1개 (목표 칼로리 대비)
    * (중앙 하단) 탄수화물/단백질/지방(g) 밸런스를 보여주는 **'수평 바 차트'** 3개
    * (하단) '아침', '점심', '저녁', '간식' 탭(Tab)으로 구분된 리스트. 각 탭 클릭 시 P-4(입력 모달) 호출.
* **[P-4: 식단 입력 모달]** '식단 입력' 기능을 다음과 같이 구체화함:
    * (상단) 음식 이름 **'자동 완성 검색창'**. (USP 반영: 배달 음식/가공식품이 최우선 검색)
    * (중단) 검색된 음식의 '1인분' 또는 '그램(g)' 단위 입력 UI.
    * (하단) '저장' 버튼.

**3.2. Back-end / 시스템 요구사항**
* **[회원]** JWT(JSON Web Token) 기반의 토큰 인증 API. (회원가입, 로그인)
* **[식단]** (USP 핵심) '음식 이름'으로 '영양 정보(칼로리, 탄/단/지)'를 반환하는 **Fuzzy Search API Endpoint**. (배달 음식 초성 검색, 오타 보정 기능 포함)
* **[계산]** 사용자가 입력한 음식 목록(log_id 기준)을 합산하여 '일일 총 섭취량' 및 '매크로 합계'를 계산하는 로직.

**3.3. 데이터 요구사항 (DB 스키마 예시)**
* `Users` 테이블: `user_id` (PK), `email` (Unique), `password_hash`, `daily_calorie_goal`
* `FoodDB` 테이블: (USP 반영) `food_id` (PK), `food_name`, `manufacturer` (예: BHC, 엽기떡볶이), `calories_per_serving`, `carbs_g`, `protein_g`, `fat_g`
* `UserFoodLog` 테이블: `log_id` (PK), `user_id` (FK), `food_id` (FK), `intake_amount` (예: 1.5인분 or 200g), `meal_type` (Enum: breakfast, lunch, dinner), `created_at` (날짜)

**4. 비기능적 요구사항 (NFR)**

* **[보안]** 모든 사용자 비밀번호는 `bcrypt` 또는 `SHA-256` 이상의 해시로 암호화하여 저장.
* **[성능]** 식단 검색 자동 완성 API 응답 속도는 평균 500ms 이내.
* **[데이터]** (USP 핵심) 킥오프 전까지 **국내 주요 배달 음식 및 가공식품 DB 최소 20,000건** 확보 필수.
* **[호환성]** Chrome, Safari 최신 2개 버전 및 모바일 브라우저 호환성 확보.
